# GlimpseAI — Filter Pipeline Modernization Specification

> **Author**: Codex implementation prompt
> **Status**: Draft implementation specification
> **Audience**: Frontend engineers, backend engineers, ML/image-processing engineers, QA
> **Scope**: Photo filter preview/save consistency, canonical filter registry, premium filter expansion, validation automation

---

## 1. Executive Summary

GlimpseAI currently exposes a rich photo filter experience, but the filter system is split across two different rendering paths:

- The frontend preview path in `artifacts/glimpse-ai/src/pages/editor.tsx` defines `FILTER_PRESETS`, `FilterState`, `cssExtra`, `buildCssFilter()`, `buildPreviewStyle()`, and `applyTransformsToBase64()`.
- The backend final-render path in `artifacts/api-server/src/lib/image-enhancer.ts` defines a separate `FILTER_PRESETS` registry built on `sharp`.
- Enhancement execution is orchestrated through `POST /api/media/enhance` in `artifacts/api-server/src/routes/media.ts`, where filter selection is currently passed through `settings.filterName`.

This architecture creates a high-risk inconsistency:

- preview output is generated by CSS/canvas approximations
- saved output is generated by `sharp` and, for some enhancement classes, the Python restoration sidecar

As a result, the user can see a preview that does not exactly match the final exported image.

This document defines the implementation specification for replacing that split architecture with a canonical, testable, production-grade filter platform.

---

## 2. Problem Statement

The modernization effort must solve the following concrete issues in the current codebase:

1. Filter definitions are duplicated across frontend and backend.
2. Preview and final rendering do not share a single source of truth.
3. Filter identity is too loosely typed and currently depends on `settings.filterName`.
4. Premium filter presentation exists, but premium-vs-standard behavior is not represented by a canonical shared contract.
5. There is no automated preview-vs-final validation framework to catch regressions.

The end state must guarantee that preview and final saved image output are visually equivalent within explicit, filter-class-specific tolerances.

---

## 3. Objectives

Deliver a system with the following outcomes:

1. A unified filter architecture that removes duplicated frontend/backend filter definitions.
2. A canonical filter registry used by UI metadata, rendering, validation, entitlement checks, and future filter additions.
3. A shared photo render pipeline for preview and export.
4. A regression suite that validates preview-to-final consistency across representative fixture images.
5. A clearly categorized standard/premium filter catalog with room for future expansion.
6. A feedback and tuning loop for iterating on filter quality without losing traceability.

---

## 4. Non-Negotiable Requirements

- Preview and final save must use the same parameter model and the same operation order.
- No filter may exist only as a frontend approximation without a canonical backend definition.
- No filter may exist only in the backend without canonical UI metadata.
- Every user-visible filter must include:
  - canonical id
  - display name
  - category
  - tier (`standard` or `premium`)
  - version
  - default parameters
  - render capabilities
  - validation thresholds
- Any preview-vs-final delta must be measurable, logged, and bounded by documented thresholds.
- Performance budgets must be defined and validated, not inferred informally.

## 5. Current Baseline Architecture

### 5.1 Frontend

Relevant implementation surface:

- `artifacts/glimpse-ai/src/pages/editor.tsx`

Current behavior:

- `FILTER_PRESETS` contains display metadata, gradient chips, `premium?: true`, `serverFilter`, and CSS approximation data.
- `buildCssFilter()` builds a browser CSS filter string.
- `buildPreviewStyle()` applies CSS filter and transform preview behavior.
- `applyTransformsToBase64()` uses `canvas` to bake transforms and CSS approximations into uploaded image data before backend enhancement in advanced mode.
- `handleProcess()` maps selected filter state into `settings.filterName` and uploads image data to the backend.

### 5.2 Backend

Relevant implementation surface:

- `artifacts/api-server/src/lib/image-enhancer.ts`
- `artifacts/api-server/src/routes/media.ts`

Current behavior:

- Backend `FILTER_PRESETS` are defined independently using `sharp`.
- Photo rendering is executed by `enhanceImage(...)`.
- Restoration and super-resolution classes may route through the Python sidecar instead of staying fully inside `sharp`.
- The backend accepts `settings.filterName`, but does not currently consume a shared canonical filter contract with typed metadata.

### 5.3 Premium Filter Baseline

Premium state is already represented in multiple places:

- frontend preset metadata via `premium?: true`
- preset API responses and DB shape via `isPremium`

This is useful, but not sufficient. Premium gating must move into the canonical filter model so UI, backend, validation, and entitlement logic all reference the same tier definition.

## 6. Target Architecture

Implement the filter platform around a single source of truth.

### 6.1 Shared Canonical Filter Registry

Create a shared workspace module, preferably under `lib/`, for example:

- `lib/filter-registry/`

This module must export:

- typed filter ids
- filter metadata
- parameter schemas
- tier metadata
- category metadata
- renderer capability descriptors
- default ordering
- validation thresholds

Each filter definition must include:

- `id`
- `version`
- `displayName`
- `category`
- `styleTags`
- `tier`
- `defaultParams`
- `intensityRange`
- `previewStrategy`
- `exportStrategy`
- `requiresSidecar`
- `deterministic`
- `validationThresholds`

The following duplicated registries must be removed or converted into adapters around the shared registry:

- frontend `FILTER_PRESETS`
- backend `FILTER_PRESETS`

### 6.2 Canonical Photo Render Pipeline

All photo filters must be rendered through the same logical sequence:

1. decode
2. color space normalization
3. crop and geometric transform
4. base tone controls
5. stylistic filter application
6. restoration/enhancement pass
7. sharpen/denoise/tone map
8. encode

This order must be defined once and enforced centrally.

### 6.3 Preview Strategy

The preferred direction is to make preview use a renderer that matches export math, such as:

- server-rendered preview through the production pipeline
- or a local/WASM/libvips-equivalent renderer using the same parameter model

If fast interactive preview must remain lightweight, implement a two-tier model:

- `fastPreview`: immediate, lower-cost response
- `canonicalPreview`: server-equivalent preview generated by the same production pipeline

If the two differ beyond threshold:

- log the delta
- surface it in internal debug tools
- fail validation tests

CSS filter strings must no longer be the authoritative preview path for filters that are exported using `sharp` or sidecar logic.

### 6.4 Export Strategy

Replace ad hoc `settings.filterName` usage with a typed canonical contract.

Minimum payload shape:

```ts
interface CanonicalFilterSelection {
  filterId: string;
  filterVersion?: string;
  tier: "standard" | "premium";
  params?: Record<string, unknown>;
}
```

The export path must record the canonical filter identity used to generate the saved result.

## 7. Library Strategy

### 7.1 Baseline Recommendation

Use `sharp` as the default production renderer unless benchmarking proves it cannot meet quality or consistency requirements.

Rationale:

- already integrated
- deterministic
- strong Node/TypeScript fit
- high performance
- suitable for a large share of correction, grading, and enhancement filters

### 7.2 Allowed Extensions

Introduce specialized processors only where they provide clear value:

- OpenCV for advanced smoothing, segmentation, or stylization gaps
- ImageMagick/libvips-compatible workflows for LUT-like or artistic transforms if `sharp` alone is insufficient
- commercial SDK only if open-source options do not meet required quality/performance and licensing is acceptable

### 7.3 Restoration Sidecar Policy

Keep the Python restoration sidecar only for filter/enhancement classes that truly require model-backed processing, such as:

- GFPGAN
- CodeFormer
- ESRGAN / Real-ESRGAN
- future model-driven restoration classes

These must be treated as a separate rendering class in the registry, with their own:

- capability flags
- determinism expectations
- validation thresholds
- latency budgets

### 7.4 Decision Record

The team must document:

- why `sharp` remains or does not remain the default
- which filters stay in `sharp`
- which filters move to specialized processors
- license implications for every added dependency
- how premium features are enforced technically

## 8. Canonical Data Contracts

Implement typed shared contracts for:

- `filterId`
- `filterVersion`
- `filterTier`
- `filterCategory`
- `filterParams`
- `rendererCapability`
- `validationThresholds`

Suggested capability descriptor:

```ts
interface RendererCapability {
  previewSupported: boolean;
  exportSupported: boolean;
  requiresSidecar: boolean;
  deterministic: boolean;
}
```

Suggested threshold descriptor:

```ts
interface ValidationThresholds {
  pixelDiffMax?: number;
  ssimMin?: number;
  phashDistanceMax?: number;
}
```

The current loose `settings.filterName` transport must be replaced or wrapped by a typed canonical filter selection object.

## 9. Filter Catalog Expansion

Audit the current catalog and classify filters into:

- correction
- cinematic / color grading
- portrait / beauty
- stylized / artistic
- restoration / upscaling

Review and explicitly decide whether to add:

- LUT-style cinematic looks
- stronger monochrome/editorial looks
- film-emulation presets
- clarity / local-contrast presets
- creator-style social presets
- portrait retouch presets with texture preservation
- advanced premium portrait and commercial looks

For each new filter, define:

- standard vs premium
- intended aesthetic
- implementation engine
- tunable parameters
- preview support
- export support
- performance cost
- validation thresholds
- sidecar fallback or degraded behavior if unavailable

The UI must visually distinguish:

- standard vs premium
- correction vs creative vs restoration
- stable vs beta/experimental, if applicable

## 10. Validation Framework

Build an automated validation suite proving preview-to-final equivalence within allowed tolerances.

### 10.1 Fixture Dataset

Create a representative photo fixture corpus covering:

- portraits across multiple skin tones
- landscapes
- high-contrast scenes
- low-light scenes
- highly saturated scenes
- soft / blurry photos
- old or damaged photos
- transparent PNGs
- JPEG, PNG, and WebP inputs
- multiple aspect ratios and resolutions

### 10.2 Validation Methods

For each filter and each fixture:

1. render preview output through the canonical preview path
2. render final output through the production export path
3. compare using one or more of:
   - exact pixel diff where deterministic equivalence is expected
   - SSIM
   - perceptual hash
   - optional histogram deltas for diagnostics

Threshold policy:

- strict thresholds for correction filters
- slightly looser perceptual thresholds for stylized filters
- separate thresholds for sidecar/model-based filters
- separate thresholds for lossy JPEG output vs lossless intermediate comparisons

### 10.3 Failure Reporting

Every validation failure must capture:

- filter id
- filter version
- fixture id
- preview artifact
- final artifact
- diff artifact or heatmap
- measured metrics
- renderer used
- pipeline version
- processor/library version
- parameter payload

### 10.4 Performance Validation

Measure and track:

- fast preview latency
- canonical preview latency
- export latency per filter class
- batch regression runtime
- memory usage
- sidecar latency and queue behavior

Suggested budgets:

- standard fast preview: low hundreds of milliseconds
- canonical preview: bounded and documented per filter class
- export latency: tracked by filter class
- regression suite: CI-friendly target runtime with artifact upload on failure

## 11. Integration Plan

### Phase 1 — Inventory and Baseline

- inventory every current filter in frontend and backend
- map frontend filter keys to backend filter ids
- identify mismatches between CSS preview approximation and backend output
- capture baseline fixtures and outputs before replacing logic

### Phase 2 — Shared Registry

- create `lib/filter-registry/` or equivalent shared package
- move filter metadata and typed ids into it
- adapt frontend and backend to consume the shared registry
- keep temporary compatibility adapters only if needed during migration

### Phase 3 — Canonical Preview

- replace or wrap `buildCssFilter()`, `buildPreviewStyle()`, and `applyTransformsToBase64()`
- align preview math with the canonical filter definitions
- if two-tier preview remains, define reconciliation and drift handling

### Phase 4 — Save/Export Alignment

- update `handleProcess()` and related frontend selection logic to send canonical filter identity and parameter payloads
- update `/api/media/enhance` to consume the typed filter contract
- ensure saved outputs retain canonical filter traceability

### Phase 5 — Validation Suite

- add automated visual regression tests
- store artifacts for failures
- enforce thresholds in CI

### Phase 6 — Catalog Expansion and Premium Distinction

- add missing premium-grade options
- unify tier metadata
- update UI labeling, badges, and grouping

### Phase 7 — Feedback Loop

- collect feedback keyed by `filterId` and `filterVersion`
- track apply/dismiss/export usage
- tune defaults by versioned iteration, not silent mutation

## 14. Acceptance Criteria

This effort is complete only when:

- every UI filter maps to one canonical registry entry
- preview and final export share the same logical filter definition
